毁经典-坦克大战

当科技的力量模糊游戏与现实的边界, 游戏将颠覆一切。

基础设置

  1. 2d 游戏的住摄像机需要设置正交性.
  2. size 直接社置到8.5, 为什么要设置到8.5呢, 因为物品的中心殿是0.5, 游戏物体的大小都是为1, 如果设置成整数的话, 生成物体是无法无边界吻合.
  3. 剩下的就是预设制作, 碰撞体、刚体、脚本添加
  4. 终于用Unity3D把这个小游戏给做完了. 总结一下入门的教程，乱七八糟的代码和资源在这里坦克大战

主角移动

移动无非就是上下左右四个

  1. 输入延迟问题
    一开始用的输入监听API是getAxis, 导致在移动的时候改变方向有延时, 因为这个API有个缓慢变化的过程, 所以后面变成getAxisRow, 这个取的是瞬时值, 问题被修复

  2. 碰撞抖动问题
    一直按住移动键的话, 移动碰到障碍时会一直抖动, 主要是因为一开始MOVE函数是放在Update函数中, 每一帧受到的力是不均匀的, 导致物体不断抖动, 直接把MOVE方法放到FixedUpdate, 这个函数是固定物理帧, 不论机器性能如何, 每一帧的时间间隔一样的, 所以当处理某些物理运动的时候, 最好把逻辑处理放在固定物理帧中。

  3. 移动优先级
    当同时按下多个键时, 选择移动方向

渲染和碰撞

渲染层级的问题

layer是对游戏中所有物体划分
如UIlayer, 3DModelLayer, smallAssetsLayer, effectLayer等。
将不同类的物体划分到不同的层，便于相机拣选，在相机的cullmask中可以选择渲染哪些层，不选择的层则不会渲染。
还可以用于射线检测对象的拣选，可以指定只对某些层的对象进行射线检测。
当把画布设置为worldspace后，画布及画布上的UI对象在世界空间默认也是靠Z值来决定渲染顺序的，离相机远的先渲染。
当设置了sorting layer后，渲染的次序就由sorting layer来确定了。

sorting layer是自定义的标识符，哪个sortinglayer在前（在layer & tags中设置的先后顺序）哪个先渲染。
order in layer是个数值，是在同一个sorting layer内的细分，sorting layer相同时order in layer的数值越小越先渲染。

屏幕空间的东西都处于UI层不需要sorting layer，只提供sort order(其实也就是order in layer)。
默认情况下sort oder都是0，此时UI物件按照在hierachy中出现的顺序决定渲染顺序。
如果sort order不同时，值越小越先渲染。

特别注意sprite默认是个3D空间的物体，就算把它放到一个屏幕空间的canvas上，它仍然是3D空间的物体，只遵循3D空间的规则：总是会被UI挡住。

同时具有sorting layer, order in layer的对象：sprite, 世界空间的canvas
仅具有sort order的对象：屏幕空间的canvas
只具有sorting layer的对象：不存在
只具有order in layer 的对象：不存在

触发碰撞的三个条件

  1. 双方都要有碰撞体
  2. 其中一方要有刚体, 最好是运动的一方
  3. 其中一方的碰撞体要是isTrigger

项目中c++热更新

c++属于编译型语言, 一旦生成可执行文件，运行逻辑不可更改, 但究其本质, 服务器本就是对数据的控制, 所以项目中采用动态加载dll的方式来进行问题的动态修复. 本质就是卸载与重新装载消息. 通过重新注册消息处理函数, 达到动态更新的目的.

游戏服务器进化史

但行好事. 莫问前程.(Do your best and God will take care of the rest.)

游戏服务器要素

  1. 运行效率,服务器性能(CPU, 内存, 网络IO)
  2. 运维部署,扩容容灾
  3. 数据存储
  4. 开发效率

第一代游戏服务器(单进程无阻赛)

client —> Server

最简单的游戏服务器, 只有一个进程, 进程退出, 游戏世界消失.
在此进程中，由于需要处理并发的客户端的数据包，因此产生了多种选择方法.

  1. 同步多线程(来一个连接便开一个线程去处理, 又分为动态申请和固定数量)

  2. 异步单线程(epoll)

  3. 异步多线程

在游戏行业中，采用Linux的epoll作为网络API，以期得到高性能，是一个常见的选择。游戏服务器进程中最常见的阻塞调用就是网路IO，因此在采用epoll之后，整个服务器进程就可能变得完全没有阻塞调用，这样只需要一个线程即可.

缺点:

1. 承载量低

###第二代游戏服务器(分区分服)

client —> Server
client —> Server
client —> Server

每个游戏服务器都是单独的一个游戏世界, 不同游戏世界的玩家要交互的话需要开辟一个公共服.随着老的服务器人数变少, 需要进行合服操作. 这样开新服、合老服的操作目前已形成一套成熟的运营手段. 分区分服虽然解决了服务器扩展的瓶颈, 但是单进程单线程方式无法充分利用服务器资源, 总会存在承载量的极限, 越复杂的游戏逻辑, 服务器的承载量越低, 因此需要突破进程的限制, 接着演变出多进程模型.

三层架构模型

集群模型

优点: 

1. 压力分散, 性能明显提升

缺点: 

1. 部署难度升高, 配置文件增多，运维难度提升, 研发成本上升

第三代游戏服务器(世界服分线)

之前有做过一个世界服的跑酷游戏, 玩家之间通过邮件交互, PK也是通过开房间实现的. 我想如果是MMO要做世界同服的话, 难度比之要大几个数量级. 对于跑酷的世界同服, 和分区分服差别不大, 数据库采用的是分库分表的处理方式, 通过玩家ID的散列值来确定他进入哪个服务器. 这种方式有个很大的缺点, 在增加和减少服务器的同时, 需要进行大量的数据迁移. 希望后续出现了MMO的世界服游戏能学习一下

开房间游戏服务器(世界服分线)

房间类玩法和MMORPG有很大的不同，在于其在线广播单元的不确定性和广播数量很小。玩家先登录大厅服务器，然后选择组队游戏的功能，服务器会通知参与的所有游戏客户端，新开一条连接到房间服务器上，这样所有参与的用户就能在房间服务器里进行游戏交互了。
