游戏服务器要素

  1. 运行效率,服务器性能(CPU, 内存, 网络IO)
  2. 运维部署,扩容容灾
  3. 数据存储
  4. 开发效率

第一代游戏服务器(单进程无阻赛)

client —> Server

最简单的游戏服务器, 只有一个进程, 进程退出, 游戏世界消失.
在此进程中，由于需要处理并发的客户端的数据包，因此产生了多种选择方法.

  1. 同步多线程(来一个连接便开一个线程去处理, 又分为动态申请和固定数量)

  2. 异步单线程(epoll)

  3. 异步多线程

在游戏行业中，采用Linux的epoll作为网络API，以期得到高性能，是一个常见的选择。游戏服务器进程中最常见的阻塞调用就是网路IO，因此在采用epoll之后，整个服务器进程就可能变得完全没有阻塞调用，这样只需要一个线程即可.

缺点:

1. 承载量低

###第二代游戏服务器(分区分服)

client —> Server

每个游戏服务器都是单独的一个游戏世界, 不同游戏世界的玩家要交互的话需要开辟一个公共服.随着老的服务器人数变少, 需要进行合服操作. 这样开新服、合老服的操作目前已形成一套成熟的运营手段. 分区分服虽然解决了服务器扩展的瓶颈, 但是单进程单线程方式无法充分利用服务器资源, 总会存在承载量的极限, 越复杂的游戏逻辑, 服务器的承载量越低, 因此需要突破进程的限制, 接着演变出多进程模型.

三层架构模型

集群模型

优点: 

1. 压力分散, 性能明显提升

缺点: 

1. 部署难度升高, 配置文件增多，运维难度提升, 研发成本上升

第三代游戏服务器(世界服分线)

之前有做过一个世界服的跑酷游戏, 玩家之间通过邮件交互, PK也是通过开房间实现的. 我想如果是MMO要做世界同服的话, 难度比之要大几个数量级. 对于跑酷的世界同服, 和分区分服差别不大, 数据库采用的是分库分表的处理方式, 通过玩家ID的散列值来确定他进入哪个服务器. 这种方式有个很大的缺点, 在增加和减少服务器的同时, 需要进行大量的数据迁移. 希望后续出现了MMO的世界服游戏能学习一下

开房间游戏服务器(世界服分线)

房间类玩法和MMORPG有很大的不同，在于其在线广播单元的不确定性和广播数量很小。玩家先登录大厅服务器，然后选择组队游戏的功能，服务器会通知参与的所有游戏客户端，新开一条连接到房间服务器上，这样所有参与的用户就能在房间服务器里进行游戏交互了。
